# BL4818 Servo Audit Notes

Date: 2026-03-26

## Scope

This note captures the current state of the firmware audit for:

- Motor-control firmware in this repository
- `flash.py`
- The custom `nu-link` programmer firmware now copied into this repository
- LDROM recovery / bootloader feasibility

It is intended to preserve the concrete findings, likely brick mechanisms, and recommended next steps so they can be revisited systematically.

## Most Likely Brick Mechanism

The strongest candidate is a packet-format mismatch between `flash.py` and the custom bridge firmware for config writes.

- The bridge firmware `nu-link/src/main.cpp` handles `CMD_UPDATE_CONFIG` by erasing the config page and writing exactly 5 raw config bytes starting at packet offset 8.
- The original `flash.py` implementation wrote config data using 32-bit packing, as though the payload were `addr + len + 5 x uint32`.
- When used against the custom bridge, the first 5 bytes actually received by the target as config data were effectively:

```text
00 00 00 00 05
```

On the MS51 family this is dangerous because it can simultaneously select LDROM boot, disable the reset pin, and set the chip into a hard-to-recover state depending on the exact device and config decode.

### Why this matters

If `CBS` is cleared while no valid LDROM image exists, the part will boot into blank LDROM after reset.

If `RPD` is also cleared, the external reset pin is disabled, which removes a major recovery path.

If the lock configuration is also altered, readback and recovery become even harder.

This is the single most plausible explanation for the observed brick.

## Fixes Already Applied In This Repository

### `flash.py`

The following fixes were already applied:

- `write_config()` now sends 5 raw config bytes to match the custom bridge firmware instead of 32-bit packed words.
- Response validation now checks both checksum and returned sequence number.
- `close()` no longer blindly runs APROM after a failed flash operation.
- `--erase` is now clearly APROM-only.
- `--chip-erase` was added for full-device erase through `CMD_ISP_MASS_ERASE`.
- APROM image size is checked against the APROM size implied by the device config.
- The tool now reports LDROM/APROM sizing when LDROM is reserved.

### Main firmware

The following correctness fixes were also applied:

- Timer1 reload now uses `FSYS` consistently instead of a stale hardcoded frequency assumption.
- RPM calculation now uses the actual clock configuration.
- Reverse speed is treated as signed instead of wrapping through an unsigned path.
- Hall-derived speed decays to zero when transitions stop arriving.
- Out-of-range protocol values for `V` and `D` commands are now rejected.

These fixes reduce obvious misbehavior, but they do not remove the major motor-risk items listed below.

## Remaining High-Risk Firmware Issues

### 1. Commutation update rate is still too low

Commutation is still updated from the 1 kHz control loop rather than directly on hall-state edges.

- Hall processing lives in `src/hall.c`
- Commutation decisions are driven through `src/motor.c`
- The loop cadence is established in `src/main.c`

With 42 hall transitions per mechanical revolution, a 1 kHz update loop only supports roughly:

```text
1000 / 42 * 60 ~= 1428 rpm
```

above which sector changes can be missed or delayed. That is still a major hardware-risk item and may cause torque discontinuities, current spikes, or loss of synchronization.

### Recommendation

- Move commutation advance onto hall interrupts or another edge-driven fast path.
- Keep the 1 kHz loop for slower control and supervision logic only.
- Re-test at low bus voltage and current-limited supply before returning to full power.

### 2. Braking / reversal path is still aggressive

The brake path still hard-shorts the motor through the low-side devices.

- `src/commutation.c` implements an all-low-sides brake state.
- `src/motor.c` uses that state during braking and reversal logic.
- `src/pwm.c` does not currently enable a hardware trip/fault braking path.

With only software current monitoring at approximately 1 kHz, this is still risky for a loaded motor or a commanded direction reversal.

### Recommendation

- Replace hard dynamic braking with a controlled decay / coast / PWM-limited brake strategy.
- Add a current threshold that immediately exits to coast or fault.
- Enable or wire a hardware shutdown path if the PWM peripheral and board support it.
- Avoid reversal under load until current limiting is proven on hardware.

### 3. Parameter page and linker are not LDROM-aware yet

The application still assumes the full APROM span is available.

- `src/flash.c` places the parameter page at a fixed top-of-flash address.
- `Makefile` still links against the full 16 KB device size.

If LDROM is reserved for a recovery bootloader, both of those assumptions become invalid.

### Recommendation

- Make APROM size an explicit build configuration.
- Derive the parameter page from `aprom_size - 128` instead of hardcoding a literal address.
- Refuse to build if the image overlaps the reserved LDROM split.

## `nu-link` Bridge Findings

The copied bridge firmware in `nu-link` still has several safety gaps that should be fixed before it is trusted as a recovery/programming tool.

### 1. No APROM bounds enforcement on program commands

`CMD_UPDATE_APROM` and `CMD_UPDATE_WHOLE_ROM` accept address and size from the host but do not enforce that the write region stays inside the valid APROM span for the current config.

This is especially dangerous once LDROM is intentionally reserved, because a host-side mistake could erase or overwrite the recovery bootloader area.

### Recommendation

- Derive APROM size from device ID plus current config.
- Reject writes with zero length, out-of-range start address, or end address beyond APROM.

### 2. Partial APROM erase loop is not page-aligned

In the unlocked `CMD_UPDATE_APROM` path, the erase loop calculates a page-aligned `start_addr` but then begins erasing from `update_addr` instead.

That makes the intended erase range inconsistent and depends on lower-level behavior to remain safe.

### Recommendation

- Start the erase loop at the aligned page base.
- Erase aligned pages only.

### 3. Config writes are not read back and validated

The bridge currently erases the config page and writes the incoming 5 bytes, then immediately acknowledges success without verifying the programmed contents.

Given that config corruption is the leading brick candidate, this deserves an explicit readback check.

### Recommendation

- Read back the 5 config bytes after programming.
- Fail the packet if readback differs from the requested bytes.

### 4. Dangerous config filtering is too weak

The existing bridge-side check only tries to block one narrow case involving reset-pin disable combined with watchdog settings, and the expression is weak enough to be error-prone.

It does not reject the more fundamental unsafe case of selecting LDROM boot while no LDROM is configured.

### Recommendation

- Always reject config writes with `RPD == 0`.
- Reject `CBS == 0` if no LDROM is reserved.
- Consider also rejecting configs that disable BOR protection during self-programming if recovery robustness is a priority.

### 5. Unknown device handling is unsafe

`get_aprom_size()` depends on flash-info lookup but does not guard against an unknown device ID before dereferencing the result.

### Recommendation

- Return `0` for unknown devices and fail writes that depend on unknown size.

## LDROM Recovery Bootloader Feasibility

An LDROM bootloader remains feasible and is the correct long-term recovery direction.

The official MS51 technical reference material documents separate boot selection and LDROM mapping behavior. A small recovery loader in LDROM would provide a hardware-independent escape path if APROM becomes invalid.

The current application image is small enough that a 1 KB or 2 KB LDROM split appears practical. A 2 KB split is the safer design point because it leaves room for:

- A minimal packet parser
- Config / APROM update support
- Integrity checks
- A clean jump into APROM

### Recommended LDROM plan

1. Reserve LDROM explicitly in config and build settings.
2. Make the application linker target the reduced APROM space.
3. Move the parameter page so it always stays inside APROM.
4. Update `flash.py` and `nu-link` to understand APROM and LDROM size dynamically.
5. Keep BOIAP / brown-out protection enabled for any in-application programming path.
6. Keep reset-pin recovery enabled.
7. Add a bootloader entry policy:
   - stay in LDROM when a GPIO strap is held
   - stay in LDROM when APROM integrity check fails
   - otherwise jump to APROM

The high-severity static findings were:

- `nu-link/src/n51_icp.c` contains two zero-return guard paths flagged as suspicious by the scanner
- `src/protocol.c` contains a comment marking the current single-digit param parser as temporary

These scanner findings are less urgent than the hardware-safety issues above, but they are worth tracking.

## Recommended Priority Order

1. Finish hardening `nu-link` so it cannot write unsafe config or overwrite LDROM accidentally.
2. Add APROM/LDROM-aware bounds enforcement everywhere in the host and bridge tooling.
3. Decide the LDROM split and make the application linker / parameter storage layout follow it.
4. Move motor commutation updates onto hall edges.
5. Rework the brake / reverse path to reduce current-spike risk.
6. Only then resume higher-energy motor testing.

## Test Recommendations Before Powering Another Motor

- Use the updated `flash.py` only.
- Use the hardened `nu-link` bridge only.
- Program a known-safe config that preserves reset-pin recovery and APROM boot.
- Verify config readback after every config write.
- Test first with current-limited supply and low duty limits.
- Avoid brake/reverse testing until commutation and current limiting are improved.
- If LDROM bootloader work proceeds, validate recovery by intentionally flashing a bad APROM image and confirming LDROM still recovers the part.
